# RunPin 작업 규칙

이 문서는 실제 작업 중 반복적으로 발생했던 문제들을 반영해 만든 규칙입니다.
모든 규칙은 예외 없이 적용합니다.

## 1. "고쳤습니다"라고 말하기 전에 반드시 검증할 것

다음 세 가지를 모두 통과하기 전까지는 수정이 끝났다고 보고하지 않는다.

1. `npx tsc --noEmit` 실행하여 타입 에러 없음을 확인
2. `git diff`로 의도한 파일이 실제로 변경되었는지 직접 확인 (편집 도구가 성공을 반환해도 실제 diff를 눈으로 재확인)
3. 변경 사항 커밋 완료 (아래 "커밋 규칙" 참조)

세 단계 중 하나라도 건너뛰고 "수정했습니다/고쳤습니다"라고 말하지 않는다.

## 2. Fast Refresh로 반영되지 않는 컴포넌트

WebView(지도, Leaflet), Modal 등은 React Native/Expo의 Fast Refresh로
변경 사항이 반영되지 않는 경우가 많다. 이런 컴포넌트를 수정했을 때는:

- 매번 "이 변경은 Fast Refresh로 확인 불가능하며, 앱을 완전히 새로 시작(reload)해야
  실제 반영 여부를 확인할 수 있습니다"라고 명시적으로 안내한다.
- 특히 `components/map/leafletHtml.ts`처럼 WebView에 주입되는 HTML/JS 문자열은
  WebView 자체를 언마운트/리마운트하거나 앱을 완전히 재시작하지 않으면 이전 버전이
  계속 표시될 수 있음을 잊지 않는다.
- 실기기/시뮬레이터에서 확인이 불가능한 환경이라면, 확인이 안 됐다는 사실을 그대로
  보고하고 "확인했습니다"라고 단정하지 않는다.

## 3. 코스/지도 좌표 작업 규칙

좌표 데이터를 새로 만들거나 수정할 때 아래를 모두 지킨다.

- **실제 도로 기반 좌표만 사용한다.** 임의로 보간하거나 직선으로 잇지 않고, 실제 도로망을
  따르는 좌표(OSM/Overpass 등에서 얻은 도로 경로)를 사용한다.
- **자기교차(self-intersection) 여부를 코드로 직접 계산해서 검증한다.** "괜찮아 보인다"는
  육안 판단이 아니라, 선분 교차 판정 스크립트를 돌려 결과를 확인한 뒤 보고한다.
- **왕복(out-and-back) 코스의 복귀 구간은 반드시 정확한 역순으로 만든다.** 좌표 배열을
  거꾸로 뒤집는 것뿐 아니라, 실제로 왕복 지점에서 자연스럽게 이어지는지 좌표 단위로
  확인한다. (이 부분에서 반복적으로 실수가 있었음.)
- **이미 검증이 끝난 기존 코스 데이터는 명시적으로 요청받지 않는 한 건드리지 않는다.**
  다른 작업을 하다가 "김에 손보는" 식의 수정 금지.

## 4. 텍스트 겹침/레이아웃 크기 문제

레이아웃, 텍스트 겹침, 크기 관련 버그는 코드를 읽고 추측으로 "이제 안 겹칠 것 같다"고
결론 내리지 않는다. 반드시 다음 중 하나로 실측한 뒤에만 완료를 보고한다.

- 콘솔 로그로 실제 렌더링된 크기/위치 값(width, height, x, y 등) 출력
- 화면에 임시 디버그 텍스트/보더를 추가해 눈으로 확인
- 스크린샷 등으로 실제 결과 확인

## 5. 커밋/푸시 규칙

- 커밋 메시지는 한국어로 작성한다.
- 의미 있는 작업 단위마다 커밋하고, 커밋 후 push까지 완료한다.
- 검증(섹션 1) 없이 커밋하지 않는다.

## 6. 절대 하지 말 것

- `.env` 파일이나 API 키가 담긴 파일을 읽거나, 그 내용을 출력/노출하지 않는다.
- 백업 브랜치 `expo-go-backup`은 삭제하거나 덮어쓰지(force-push, reset 등) 않는다.

## 7. 지도 라이브러리 관련 이력

현재 지도는 **WebView + Leaflet** 조합을 사용 중이다. 과거 `react-native-maps`로
전환을 시도했으나 iOS에서 디자인 커스터마이징 한계에 부딪혀 롤백한 이력이 있다.
`react-native-maps`(또는 다른 지도 라이브러리)로의 전환을 다시 제안하기 전에는
먼저 사용자에게 확인을 받는다. 임의로 전환 작업을 진행하지 않는다.

## 8. 멀티 기기(Windows/맥) 작업 규칙

동일한 저장소를 Windows와 맥 등 여러 기기에서 오갈 때는 아래를 모두 지킨다.

- 작업 시작 전에는 항상 `git pull`로 최신 상태를 받아온 뒤 작업을 시작한다.
- 작업 단위가 끝나면 커밋 후 push까지 완료한 다음 세션을 종료한다 (섹션 5 "커밋/푸시 규칙" 참조).
- 기기를 바꾸기 전에는 `git status`로 커밋되지 않은 변경사항이 없는지 반드시 확인한다.
- `.env` 파일은 `.gitignore`에 포함되어 있어 기기 간 자동으로 동기화되지 않는다. API 키를
  새로 발급하거나 변경했을 때는 다른 기기의 `.env`도 수동으로 동일하게 업데이트해야
  한다는 점을 사용자에게 알린다.
- 동일 시점에 두 기기에서 동시에 작업하지 않는다 (충돌 방지).
