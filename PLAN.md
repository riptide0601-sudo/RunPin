# RunPin — 개발 계획

> GPS 기반 러닝 코스 추천 & 근거리 러너 매칭 서비스
> 개인 기록 중심의 기존 러닝 앱과 달리, **코스 발견**과 **커뮤니티 연결**에 방점.

---

## 1. 프로젝트 개요

**한 줄 정의:** 새로 이사한 동네에도 검증된 러닝 코스를 발견하고, 근처에서 비슷한 페이스로 뛰는 러너와 실시간으로 연결되는 러닝 앱.

**기획 출발점:** "우리 동네에는 마땅히 뛸 곳이 없는데, 다른 사람들은 대체 어느 코스로 달리고 있을까?"

### 핵심 가치 제안 (3축)
| 가치 | 설명 |
|---|---|
| 코스 자동 발견 | 검색이 아닌 자동 제안으로 동네 생활 코스 확보 |
| 실시간 러너 매칭 | 반경 내 비슷한 페이스의 러너를 지오펜싱으로 즉석 매칭 |
| 기록 부담 최소화 | 업로드/정리 마찰을 없애 자연스러운 데이터 축적 유도 |

---

## 2. 문제 정의 (Problem)

- 검색해도 나오는 건 등록된 공식 코스 몇 개뿐
- 실제 동네 러너들이 자주 뛰는 "생활 코스" 정보는 구할 방법이 없음
- 기존 앱(A, B 어플)은 개인 기록 중심 → 코스 발견·커뮤니티 매칭 기능 부재

---

## 3. 타겟 유저 (Persona)

### Persona A — 서준, 29세 · 러닝 8개월차
- 이사 온 지 3개월, 동네 코스 정보 전무 → 매번 같은 루트 반복
- **Needs:** 검증된 동네 코스 발견
- **Pain:** 검색해도 정보 부재

### Persona B — 지현, 33세 · 러닝 2년차
- 같은 코스만 뛰어 흥미 저하, 페이스 맞는 러닝 메이트를 오프라인에서 찾기 어려움
- **Needs:** 새 코스 + 러닝 메이트
- **Pain:** 매칭 채널 부재

| 기능 | A 어플 | B 어플 | RunPin |
|---|---|---|---|
| 코스 추천 | ✗ | ✗ | ✓ |
| 커뮤니티 매칭 | ✗ | ✓ | ✓ |
| 실시간 위치 매칭 | ✗ | ✗ | ✓ |
| 선제적 저장 제안 | ✗ | ✗ | ✓ |

**포지셔닝 맵:** 능동적 × 커뮤니티 중심 사분면 — 경쟁사 대비 가장 능동적이고 커뮤니티 중심적인 위치를 지향.

---

## 5. 리서치 근거

- 심층 인터뷰 6건 / 설문 응답 42건 / 리서치 기간 2주

**핵심 인사이트**
- 커뮤니티: "근처에 비슷한 페이스로 뛰는 사람 있으면 같이 뛰고 싶음" / "러닝크루는 진입장벽이 높음"
- 업로드: "굳이 정리해서 올릴 이유를 못 느낌" / "앱이 먼저 물어봐준다면 부담이 훨씬 덜함"
- 동기부여: "내가 올린 코스가 남에게 도움 되면 신경 써서 올릴 것" / "랭킹·뱃지 보상이 있으면 동기부여됨"

**적용 이론**
- 디폴트 옵션 효과 → 선택을 요구하지 않고 기본값 제시 (선제적 코스 제안)
- 마찰 감소 원칙 → 행동 단계 수 최소화 (원탭 승인)
- 사회적 증거 → 타인 행동 가시화로 참여 확신 강화 (코스 랭킹, 매칭 표시)

---

## 6. 악순환 구조 (The Loop) — 반드시 끊어야 할 지점

```
업로드 저조 → 코스 데이터 부족 → 추천 품질 저하 → 참여 유인 저하 → (다시 업로드 저조)
```

→ **선제적 코스 제안(Feature 01)**으로 최초 진입점을 낮춰 루프를 역방향으로 전환하는 것이 최우선 과제.

---

## 7. 기능 명세 (Feature Spec)

### Feature 01 — 선제적 코스 제안 (우선순위 1위, 92점)
- 러닝 종료 시 앱이 먼저 저장 여부를 질문 (사용자는 승인만)
- Flow: 러닝 종료 → 원탭 승인 팝업 → 자동 카테고리화
- **목표:** 코스 업로드 전환율 25%

### Feature 02 — 궤적 매칭 모델 (우선순위 4위, 60점 · 기술 난이도 최고)
- 동일 코스라도 진입 각도·정지 구간·이탈 지점이 사람마다 달라 궤적이 제각각
- GPS 궤적 유사도 기반으로 부분 구간까지 자동 식별 필요
- Flow: GPS 원본 궤적 → 노이즈 정규화 → 구간 유사도 계산 → 부분 구간 매칭
- 고도화 단계로 별도 분리 (초기 MVP 범위 아님)

### Feature 03 — 지오펜싱 러너 매칭 [메인 기능] (우선순위 2위, 78점 · 리서치 최다 요구)
- 반경 내 비슷한 페이스의 러너를 실시간 감지 → 즉석 매칭 제안
- 구성 요소: 반경 설정 / 러닝 페이스 필터링 / 실시간 알림 / 동반 러닝
- UI: 유저 = 빨간 점, 페이스 유사 매칭 대상 = 파란 점
- 부가: 기록 비교, 혜택(리워드) 연동
- **목표:** 지오펜싱 매칭 성사율 30%

### Feature 04 — 커뮤니티 활성화 루프 (우선순위 3위, 70점 · 선행 데이터 필요)
- Flow: 함께 뛴 기록 비교 → 코스 랭킹 진입 → 뱃지·리워드 지급 → 업로드 유인 재생산
- **목표:** 주간 재방문율 상승 15%

## 8. 개발 우선순위 로드맵

| 순위 | Feature | 점수 | 이유 |
|---|---|---|---|
| 1 | 선제적 코스 제안 | 92 | 낮은 난이도, 데이터 축적 시작점 |
| 2 | 지오펜싱 러너 매칭 | 78 | 리서치에서 가장 많이 요구된 기능, 핵심 차별화 |
| 3 | 커뮤니티 활성화 루프 | 70 | Feature 1·2의 선행 데이터 필요 |
| 4 | 궤적 매칭 모델 | 60 | 기술 난이도 최고, 별도 고도화 단계로 분리 |

**권장 개발 순서:** MVP는 Feature 01 → Feature 03 순으로 진행하고, Feature 04는 유저 데이터가 쌓인 후, Feature 02는 별도 R&D 트랙으로 후순위 진행.

## 10. 기술적 고려사항 (논의 필요)

- **플랫폼:** 모바일 앱 (GPS 상시 추적 필요 → React Native/Expo 또는 네이티브 검토)
- **위치/GPS:** 백그라운드 위치 추적, 배터리 소모 최적화 필요
- **실시간 매칭:** 지오펜싱 + 실시간 알림 → WebSocket 또는 실시간 DB(Firebase Realtime DB/Firestore) 후보
- **궤적 매칭 알고리즘:** GPS 노이즈 정규화 + 구간 유사도 계산 → 별도 알고리즘/ML 검토 필요 (Feature 02, 후순위)
- **인증:** 초기 로그인/인증 방식 결정 필요 (예: Firebase Auth)
- **데이터 모델 초안 필요 항목:**
  - User (프로필, 페이스 이력)
  - Course (좌표 시퀀스, 카테고리, 난이도, 거리)
  - RunLog (GPS 궤적, 시간, 페이스)
  - Match (매칭 상태, 참여자, 반경)
